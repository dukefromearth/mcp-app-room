# ADR: Core-First MCP Support Plan for roomd SDK (Q2-Q3 2026)

- Date: 2026-02-28
- Status: Proposed
- Decision Owners: roomd maintainers

## Context

`roomd` is currently optimized around room orchestration and MCP Apps-style UI mounting. It already proxies several MCP server primitives (tools/resources/prompts list + tool call) but it is not yet a full protocol client surface for all non-experimental MCP features.

Current implementation signals:

- Strong room lifecycle/state/idempotency model in `RoomStore`.
- MCP server connectivity via streamable HTTP with SSE fallback.
- UI-first mount flow and inspection endpoint.
- Partial server primitive coverage (missing some methods and notifications).

The goal for the next two quarters is to move from "works for UI-centric servers" to "first-class MCP host/client behavior," while keeping implementation simple and adapter-driven.

## Decision Summary

1. `roomd` will adopt a core-first architecture: strict MCP base protocol + lifecycle + capability negotiation first, then capability modules.
2. `roomd` will treat all MCP extensions (including MCP Apps) as opt-in adapters, not core assumptions.
3. `roomd` will target a Tier 2 conformance bar by end of Q2 2026 and Tier 1 by end of Q3 2026 for non-experimental protocol features.
4. UI resources will remain optional for mount/use paths; non-UI MCP servers must be first-class citizens.
5. `roomd` will publish two explicit support profiles:
   - Core MCP Profile (required for v1).
   - MCP Apps Profile (`io.modelcontextprotocol/ui`) as optional extension behavior.

## MCP Apps Normative Anchors

The Apps profile decisions in this ADR are based on these constraints from the Apps spec:

1. MCP Apps is an optional extension negotiated via capabilities (`io.modelcontextprotocol/ui`), not a core requirement (`/Users/duke/Documents/github/ext-apps/specification/2026-01-26/apps.mdx:53`, `:1494`).
2. Tool-to-UI linkage is optional at tool metadata level (`resourceUri?`), so not all tools/mounts can require UI (`/Users/duke/Documents/github/ext-apps/specification/2026-01-26/apps.mdx:326`).
3. If UI metadata is absent or client does not support Apps, behavior falls back to standard text tool behavior (`/Users/duke/Documents/github/ext-apps/specification/2026-01-26/apps.mdx:389`).
4. Hosts must fetch referenced UI resources via `resources/read`, but servers may omit UI-only resources from `resources/list` (`/Users/duke/Documents/github/ext-apps/specification/2026-01-26/apps.mdx:391`, `:393`).
5. Web-host rendering requires sandbox-proxy origin separation and CSP/permission enforcement (`/Users/duke/Documents/github/ext-apps/specification/2026-01-26/apps.mdx:472`, `:479`).
6. UI resources should be predeclared and referenced from tool metadata when Apps is used (`/Users/duke/Documents/github/ext-apps/specification/2026-01-26/apps.mdx:1586`).

## Answers to Key Questions

### 1) What is required for v1 "MCP support"?

For `roomd` v1, required means:

- Base protocol + lifecycle compliance (`JSON-RPC`, initialization handshake, negotiated capabilities, shutdown discipline).
- Full support for non-experimental core primitives as applicable to a host/client role.
- Capability-gated behavior for optional features; no hard assumptions that a server has tools/resources/prompts/UI.

Evidence:

- Base + lifecycle are mandatory; other components may be implemented per needs.
- Specification revisioning and capability negotiation are normative.

### 2) Are extensions required for conformance?

No. Extensions are explicitly optional and disabled by default unless both sides opt in. SDKs choose which extensions to implement.

Decision implication:

- MCP Apps remains supported via adapter.
- Core roadmap does not block on extension coverage.

### 2a) How should MCP Apps specifically fit this architecture?

- MCP Apps is an optional extension and must be explicitly negotiated.
- When negotiated, host behavior must follow MCP Apps rules for tool metadata linkage, UI resource read, and sandbox lifecycle.
- When not negotiated, behavior must gracefully degrade to standard MCP (text-first tool results).

Decision implication:

- `roomd` keeps UI behaviors inside an Apps adapter and never in core mount/session semantics.

### 3) What compatibility policy should roomd follow?

- Primary target protocol: `2025-11-25` (current).
- Preserve ability to connect to older HTTP+SSE servers as compatibility mode.
- Keep transport fallback explicit and testable.

### 4) What transport baseline should roomd use?

- Must support streamable HTTP.
- Should support stdio "whenever possible".
- Custom transports are allowed, but must preserve lifecycle + JSON-RPC semantics.

Decision implication:

- Q2: harden streamable HTTP + legacy SSE.
- Q3: add stdio transport adapter for local servers.

### 5) How should absent capabilities be handled?

- Capability absence must produce graceful degradation, not hard failure.
- `roomd` should avoid calling methods for undeclared capabilities and return typed "unsupported capability" errors to its callers.

### 6) What is expected for conformance measurement?

- MCP now uses automated conformance tests and tiering.
- Tier 2 requires 80% applicable conformance; Tier 1 requires 100%.
- Experimental features and extension tests are excluded from tier requirements unless claimed.

Decision implication:

- `roomd` CI should track conformance score as a release gate.

### 7) Are tasks required for v1?

No. Tasks are explicitly experimental in `2025-11-25`.

Decision implication:

- Keep tasks behind a feature flag and out of v1 critical path.

### 8) Is HTTP auth mandatory?

Authorization is optional at protocol level, but HTTP-based implementations should follow MCP authorization when supported.

Decision implication:

- Q2 adds pluggable auth strategy interfaces.
- Q3 adds production-ready HTTP auth flow integration and security hardening.

### 9) What retry/idempotency posture should roomd use?

- Preserve roomd command idempotency guarantees.
- For upstream MCP operations, treat transport disconnects as retryable only for safe/idempotent operations unless a caller explicitly opts into replay for mutating requests.
- Apply request timeouts and cancellation notifications consistently.

### 10) How should cross-server boundaries be enforced?

- Keep one isolated MCP session per mounted server instance by default.
- Do not allow implicit cross-server tool delegation.
- Keep server-specific capability and auth context scoped to that mount/session.

### 11) What error semantics should roomd expose?

- Preserve upstream JSON-RPC protocol errors where possible.
- Add roomd-level typed errors for routing/policy failures (`UNSUPPORTED_CAPABILITY`, `AUTH_REQUIRED`, etc.).
- Keep these error codes stable across CLI/API surfaces to protect SDK users from churn.

### 12) Should roomd require legacy transport support forever?

No. Legacy HTTP+SSE support is compatibility behavior, not a permanent core guarantee.

Decision implication:

- Keep fallback in the near term.
- Introduce a deprecation policy and sunset date once streamable HTTP adoption is sufficient.

### 13) How should roomd handle MCP Apps discovery and fallback?

- Prefer `_meta.ui.resourceUri` from tools for UI linkage.
- Treat `resources/list` as advisory for Apps discovery because UI-only resources may be omitted.
- If no negotiated Apps support or no UI metadata is available, continue with non-UI mount/use behavior.

### 14) What security stance should roomd use for Apps rendering?

- Keep sandbox-proxy architecture with strict separation of host and sandbox origins.
- Enforce CSP/permission metadata when present and restrictive defaults when absent.
- Never allow Apps messaging to bypass capability/policy checks.

## Current roomd Gaps vs Core-First Goal

1. Feature coverage gaps:
- Missing prompt retrieval path (`prompts/get`) in roomd API.
- No completion API exposure.
- No resource subscribe/unsubscribe endpoints.
- No explicit handling of server-initiated notifications beyond current room event model.

2. Capability modeling gaps:
- No explicit capability matrix persisted per mounted server session for routing/guardrails.
- Commands and store model still encode UI-centric assumptions for mounts.

3. Transport/auth hardening gaps:
- No stdio adapter.
- No explicit protocol-version/capability assertions at roomd boundary layer.
- No roomd-level auth policy abstraction for upstream MCP HTTP servers.

4. Conformance operations gap:
- No conformance suite integration in CI as an objective release gate.

5. MCP Apps layering gap:
- Apps-specific logic currently leaks into core mount assumptions.
- Server inspection path assumes resource enumeration is always available.
- No explicit negotiated extension state is persisted per mount/session.

## Architecture for the Next Two Quarters

## A. Capability-First Adapter Layers

`roomd` should split MCP integration into stable layers:

1. Transport adapter (`streamable-http`, `legacy-sse`, `stdio`)
2. Session core (initialize/version/capabilities/timeout/cancellation/progress)
3. Capability modules:
   - Server primitives: tools/resources/prompts/completion/logging
   - Client primitives: roots/sampling/elicitation
4. Extension adapters:
   - `io.modelcontextprotocol/ui` (MCP Apps)
   - future extensions behind feature flags

## B. Room Model Contract Changes

1. Mount must be protocol-first:
- A mount represents an MCP server session, not a UI instance.
- UI linkage is optional metadata if extension negotiation supports it.

2. Session metadata per mount:
- negotiated protocol version
- negotiated capabilities
- declared extension settings
- transport type and auth mode

3. Command/API behavior:
- Route requests by negotiated capabilities.
- Return stable typed errors for unsupported operations.

## C. Error and Degradation Model

`roomd` should normalize errors into:

- `UNSUPPORTED_CAPABILITY`
- `UNSUPPORTED_EXTENSION`
- `TRANSPORT_UNAVAILABLE`
- `AUTH_REQUIRED` / `AUTH_FAILED`
- pass-through MCP protocol errors when applicable

This keeps roomd callers decoupled from transport-specific failures.

## D. Testing and Operational Discipline

1. Add conformance CI stage (non-experimental required tests).
2. Add transport matrix tests (streamable, fallback SSE, stdio when added).
3. Add capability-negotiation contract tests for all API routes.
4. Add chaos tests for reconnect/timeouts/cancellation behavior.

## E. MCP Apps Extension Profile

When `io.modelcontextprotocol/ui` is negotiated, `roomd` applies these rules:

1. Discovery:
- Parse tool metadata from `_meta.ui.resourceUri` (with temporary compatibility for deprecated flat key where needed).
- Do not hard-fail on `resources/list` failure during inspection; UI resources may be omitted from listing.
- Resolve UI with best-effort strategy and return warnings/hints, not fatal errors, when core operations can proceed.

2. Fallback:
- If negotiated Apps support is absent, tools remain callable and return text/structured content as normal MCP.
- Mount/session creation must not require a UI URI.

3. Visibility and access control:
- Preserve tool visibility semantics for model/app audiences when hosting app-driven tools.
- Prevent implicit cross-server app tool calls.

4. Sandbox and rendering:
- Maintain host/sandbox origin separation.
- Apply CSP/resource-domain restrictions and restrictive defaults.
- Gate message forwarding by policy and negotiated capabilities.

## Roadmap (Two Quarters)

## Q2 2026 (Mar-May): Tier 2 Foundation

1. Introduce capability matrix per session and enforce routing guards.
2. Fill core server primitive gaps: add `prompts/get`, completion, and resource subscription endpoints.
3. Make UI extension entirely optional in mount semantics.
4. Add typed error taxonomy and API response stability guarantees.
5. Integrate conformance tests in CI; target >=80% applicable pass rate.
6. Refactor inspection/mount flow to be Apps-aware but non-Apps-safe:
   - `resources/list` best-effort
   - metadata-first UI discovery
   - warning surfaces instead of hard failures when core operations remain valid

Exit criteria:

- No API path assumes UI extension presence.
- Capability-negotiation tests pass.
- Tier 2 conformance threshold reached in CI.

## Q3 2026 (Jun-Aug): Tier 1 Hardening

1. Add stdio transport adapter.
2. Add client-feature support modules (roots/sampling/elicitation) where roomd acts as MCP client host.
3. Implement HTTP auth strategy abstraction + secure defaults for remote servers.
4. Close remaining non-experimental conformance gaps; target 100% applicable pass rate.
5. Publish compatibility/deprecation policy and extension support matrix.
6. Publish explicit Core vs Apps support matrices with contract tests for both profiles.

Exit criteria:

- Tier 1 conformance target met for non-experimental required tests.
- Stable v1 contract for transport/capability/error behaviors documented.

## Non-Goals for This ADR

1. Making experimental tasks mandatory for v1.
2. Requiring any specific extension for v1 conformance.
3. Replacing roomd’s room orchestration model.

## Risks and Mitigations

1. Risk: over-expansion into extension work.
- Mitigation: hard gate extension work behind core conformance milestones.

2. Risk: UI-first assumptions re-enter through feature work.
- Mitigation: enforce "capability present or fail typed" checks in shared middleware.

3. Risk: transport complexity causes regressions.
- Mitigation: adapter isolation + transport matrix tests + strict integration fixtures.
